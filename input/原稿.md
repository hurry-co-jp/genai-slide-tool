# はじめに
今までにもスライド作成AIはありましたが、Geminiのスライド生成対応、Nano Banana ProおよびNotebookLMのスライド生成機能が登場したことでかなり雑な指示でも簡単にそれっぽいスライドを生成できるようになりました。
しかし、何度かAIに生成させたものを見ていると「なんか既視感あるな、AIっぽいな」という感覚になってきます。
また、同じ内容で生成させても毎回異なる内容、異なるデザインで生成されるため、文章や画像以上にいい感じのものを生成してもらうことが難しいです。
これでは簡易的なその場しのぎのスライドとしては通用しても、ちゃんとした発表で使うスライドとしては流石に厳しいものがあります。
そこで同様のことに課題意識を持った人たちが多くの工夫を考えていますが、中でもとくに再現性が高くて汎用性があると思えたのが以下のような「情報を構造化して与える」というアプローチです。

https://note.com/yoshifujidesign/n/n7412bccb5762
https://note.com/samuraijuku_biz/n/ncace2a0775ac

私は今までもAIと対話していてある程度考えがまとまったら一旦MarkdownでまとめさせたりMermaid形式のフローチャートなどで整理させたりしていたのですが、アウトプットにランダム性のあるAIに対してはこのような構造化された形式で出力を行わせると、格段に自分とAIとの認識を一致させやすくなると感じています。
ちょっと認識が違うときは出力されたものを自分で修正して渡すことで、AIに対してだいぶ正確に意図通りの修正を行わせることもできます。
同様のやり方をスライド生成に対しても行えば、当然スライドも格段に自分の意図通りのものを生成させやすくなると考えられます。
そこで今回はエンジニアリングの観点からプロセスを分解し、適切に制御することで効率的に実用的なスライドを生成させることを目指します。

# スライド作成のプロセスを分解する
まずは人間がスライドを作成するときにどんなことをやっているのか整理してみます。

![](./ignore/phase.png)

## 着想・理解や考えの解像度をあげる
まず、最初は「スライドを作ろう」というキッカケで始まるはずです。
何かしらをインプットとして誰かに何かを伝えたいと思うところがスタートになります。
しかし、この時点ではまだ全体的に漠然としていて「なんとなく」の部分が多いと思います。
この「誰（ターゲット）」に「何（コアメッセージ）」を伝えたいと自分は考えているのかを深堀りし、解像度を深めていくことが最初のフェーズになります。
ここでのゴールは全体を通して一貫した軸となる考えを確立させることになります。

## 言語化
次に、1で確立させた軸（ターゲット、コアメッセージなど）をもとに、ストーリーを考えていきます。
ターゲットに対してコアメッセージが正確に伝わるように届けるにはどういう話をしていけばいいか、納得させるためにどんな前提や根拠が必要か、話の流れはどう進めるのがいいのかなど、スライドのプロットを作成します。
本当に急ぐときや簡易的な発表でいいときは、このプロットを簡単に箇条書きでまとめてシンプルなスライドにして発表することもよくあります。

## 視覚化
最後に、作成したプロットをPowerPoint等を使ってキレイなスライドに起こしていきます。
ここが一般的に想像する「スライドを作成する」に相当する部分になりますが、実際は上記の通り、いきなりスライドを作成するようなことはせず、この時点で内容はほぼ完成している状態になります。
なお、「スライドを作成する」を文字通り「PowerPoint等でスライドを作るだけ」だと考えて上記の1や2を飛ばしてしまうと、目的と内容とデザインをすべて同時に考えながら作業することになり、何が言いたいのかわからないスライドができてしまう可能性が高まります。
もしかしたら慣れている人はいきなり作業を始めても高品質なスライドを作れるのかもしれませんが、そのような人はおそらく1や2で行うようなことを頭の中だけでやれてしまっているのだと思います。
これらのプロセスを整理すると以下のようになります。

| フェーズ | Start (入力) | 具体的なタスク (処理) | Goal (出力) |
| --- | --- | --- | --- |
| 1. 着想 | 断片的な情報  (モヤモヤした気持ち) | ・情報収集  ・情報の構造化  ・核（コア）の抽出 | ターゲット、コアメッセージ |
| 2. 言語化 | 核となる想い  (論理なし) | ・ターゲット分析  ・ストーリー構成  ・全ページの文章作成 | 文字だけの原稿  (字コンテ) |
| 3. 視覚化 | 文字原稿  (読みづらい) | ・デザイン整頓  ・レイアウト配置  ・図解への変換 | 完成版のスライド  (直感的にわかる) |

# 各フェーズでのAI活用方法を考える

## Phase 1 & 2：着想と言語化でのAI活用
ここはすでに多くの方が実践されている手法かと思いますので、要点のみを記します。
重要なことは、「デザイン（Phase 3）に入る前にストーリーをほぼ100%固めておくこと」です。

### Phase 1: 着想（軸を決める）
ここでのAIは「壁打ち相手」として、自分の考えの解像度をあげていくために対話していきます。
「情報の拡散（トレンドの列挙など）」や「収束（グルーピング）」を行わせ、最終的に「一言でいうとどういう人に何が言いたいのか（コアメッセージ）」を決定します。

### Phase 2: 言語化（脚本を作る）
ここでのAIは「編集者」として、軸を補強していくようにテキストのみで論理構造を組み立てて原稿を作成させていきます。
ターゲット（ペルソナ）を設定した上で構成案を作らせ、各スライドで入れたい文言やポイントを作り込んでいき、Markdownの原稿作成を目指します。
Mermaidでのフローチャートなどの図、言いたいことを表現した画像なども作っておくといいです。
図や画像がある場合は、ファイルをバラして置いておくよりPDF化して1つのファイルとしてしまった方がいいと思います。

## Phase 3: 視覚化
今回の記事の本題です。
完成した原稿をAIに渡し、「これをいい感じのスライドにして」と依頼すると、原稿を作成せずに作成したスライドより格段に品質が高くて意図通りのスライドが生成されるようになります。自分の意図を反映して作り込んだ原稿を用意したことで再現性が高められています。
しかし、何度か生成を行っていると、生成するたびにレイアウトが変わったり、デザインに全体的な統一感がなかったり、AI特有のイラストやアイコンなどの表現が頻出したりなどが気になってきます。もちろん何回も生成してみていい感じのものだけを切り貼りして使うといったデザインガチャのように使うこともできますが、あまりにも運任せすぎてそのままでは実用レベルには達しません。
この「デザインガチャ」をなるべく回避するためにはプロンプトを作り込むことが有効ですが、それだけで終わらせてしまうとプロンプト集を用意するなどの個人の工夫になってしまいます。
誰がやっても再現性の高い「仕組み」とするために、ソフトウェア開発の手法にならってプロセスを整理して組み立てるエンジニアリングを行いたいと思います。

### デザインフェーズを設計と実装のステップに分ける
ここまでで、いきなりスライド作成を始めるのではなく先に言語化をするべきだという話をしました。
これと同じことをデザインフェーズに対しても考えます。
つまり、「デザインの言語化」と「デザインの実装」の2段階に分けます。
ただし、デザインを言語化するための準備として「デザインテンプレートの言語化」も行っておきます。これも含めると3段階になりますが、これは一度作れば今後使い回せる資産となりますし、他の人が作ったものを使わせてもらうのでもいいと思います。
フロントの作成でも通常はいきなりReactやCSS書いたりしないで、一般的にどんな表現（テンプレ）があるのかを踏まえながらどういう情報をどこにどういう見せ方で配置しようとか考えるはずで、スライド作成の場合もそれと同様のプロセスで考えます。
これによって、「実際に画面を触りながらなんとなくいい感じのデザインを考える」から「ここはこういうデザインにしたいを考える→実装方法を考える」となり、ステップを踏んで作成できるようになります。
とはいえ、どう工夫してもAIに100%の完成度を求めるのは現実的でないため、ここでは70%～90%程度の完成度のものを安定的に生成してくれるようになることを目指し、最後に人間が微調整をするというHuman-in-the-Loopのプロセスで考えます。

### Step 0: デザインテンプレートを言語化する
まずはデザインのベースとなるテンプレートを作っておきます。
このテンプレートを作っておくことで、デザインを決めるときにどのスライドがどのテンプレートをベースにするのかを指定するだけでだいたい意図通りで統一感のあるスライドが生成できると考えられます。
テンプレートをどのように表現するのがよいかは検討の余地がありますが、私は今のところ大まかに以下の2つが必要だと思っています。

- 全体的なデザイン定義書
- パターンごとのレイアウト定義書

#### 全体的なデザイン定義書
いわゆるトンマナを定義しておくものです。
白を基調とした青っぽいデザイン、タイトルは48pt、フォントはメイリオ、イラストやアイコンはフラットでシンプルなもの
のように、スライド全体を通して一貫させたい情報を記載しておきます。
自然言語でもいいと思いますが、YAMLなどの構造化されたデータとして作っておくのが拡張させやすくていいと思っています。
AIに自然言語で「こんな感じ」を伝えてYAMLを生成してもらえばいい感じのものができるので、これを成長させていけばいいです。
これだけでも割と完成度の高いスライドが生成されるようになります。

デザイン定義書の作成例

```yaml
colors:
  # ベースカラー（背景色）
  base_background: "#FFFFFF" # 完全な白
  base_text: "#333333"       # 視認性の高いダークグレー（真っ黒ではない）

  # メインカラー（企業のブランドカラー、信頼・知性）
  primary: "#0052CC"         # 鮮やかながら落ち着きのあるテックブルー
  primary_light: "#DEEBFF"   # アクセントや背景装飾用の薄いブルー

  # アクセントカラー（強調、ポイント）
  accent: "#00B8D9"          # 少し緑寄りの明るいシアン（先進性、クリーン）

typography:
  # フォントファミリー（OS標準のゴシック体を優先指定）
  font_family_ja: "Noto Sans JP, Hiragino Kaku Gothic ProN, Meiryo, sans-serif"
  font_family_en: "Roboto, Helvetica Neue, Arial, sans-serif"

  # 文字サイズと太さのルール
  title_slide:
    main_title: { size: 48pt, weight: "bold", color: "#333333" }
    sub_title:  { size: 24pt, weight: "normal", color: "#0052CC" }
  body_slide:
    page_title: { size: 28pt, weight: "bold", color: "#333333" }
    heading1:   { size: 20pt, weight: "bold", color: "#0052CC" } # 大見出し
    body_text:  { size: 16pt, weight: "normal", color: "#333333", line_height: 1.5 }
    small_text: { size: 12pt, weight: "normal", color: "#666666" } # フッターなど
```

#### パターンごとのレイアウト定義書
スライドのレイアウトはそのページで見せたいことに応じていくつかのパターンがあると考えられます。
例えば「2カラムで左右を対比的に表現する」、「左から右に流れていくフローを表現する」、「ピラミッドストラクチャーで表現する」などのようなものです。
これもYAMLでパターンのリストとそれぞれのパターンがどんなものなのかを定義しておくといいと思っていますが、リファレンスとしてPDFや画像で実際にサンプルも用意しておいた方が完成度が高くなるのではないかと思っています。

レイアウト定義書の作成例

```yaml
# --- カテゴリS: 構造・基本 (Structure & Basic) ---
- id: S01
  name: タイトルスライド（表紙）
  category: structure_basic
  image_name: S01_title_cover.png
  description: プレゼンテーションの表紙となるレイアウト。最も大きなフォントでタイトルを配置し、その下にサブタイトル、日付、所属、氏名などを配置する標準的な構成。
  usage_hints:
    - プレゼンテーションの開始（1枚目）に必ず使用する
    - タイトルを最も強調し、聴衆の期待感を高めたい

# --- カテゴリA: 比較・対比 (Contrast & Comparison) ---
- id: A01
  name: 左右対比型（Before/After）
  category: contrast
  image_name: A01_contrast_before_after.png
  description: 画面を中央で左右に分割し、対立する概念や変化の前後を比較するレイアウト。中央に変化を示す矢印を配置することが多い。
  usage_hints:
    - 現状（As Is）と理想（To Be）のギャップを示したい
    - 課題とそれに対する解決策を対比させたい
    - 導入前と導入後の変化を強調したい

- id: A02
  name: 3点比較表型（スペック）
  category: contrast
  image_name: A02_contrast_table_3cols.png
  description: 縦軸に比較項目、横軸に3つの対象（自社・他社A・他社Bなど）を並べたマトリクス表形式のレイアウト。
  usage_hints:
    - 複数のプラン（松竹梅）を比較提示したい
    - 競合他社に対する自社製品の優位性を機能別に示したい
    - 複数の選択肢のメリット・デメリットを一覧化したい

# --- カテゴリB: プロセス・時系列 (Process & Timeline) ---
- id: B01
  name: 横並びステップ型（フロー）
  category: process
  image_name: B01_process_steps_horizontal.png
  description: 左から右へと順序立てて進む手順や段階を、矢印で繋がった3〜4つのボックスで表現するレイアウト。
  usage_hints:
    - サービスの導入から運用開始までの手順を説明したい
    - 業務フローやデータの流れを示したい
    - PDCAサイクルのような循環する流れを示したい

# --- カテゴリC: 構造・構成要素 (Structure & Components) ---
- id: C02
  name: ピラミッド階層型
  category: structure
  image_name: C02_structure_pyramid.png
  description: 上下関係や包含関係を、三角形の層を積み重ねたピラミッド構造で表現するレイアウト。
  usage_hints:
    - 企業のビジョン・ミッション・バリューの階層を示したい
    - 戦略・戦術・アクションプランのレベル感を整理したい
    - 組織構造や体制図の上下関係を示したい
```

### Step 1: デザインを言語化する
ここではAIに「デザイナー」として、この内容だったらこんな見せ方をするのがいいということを考えてもらいます。
ここまでで以下のインプットが揃います。

- 原稿.md
- 原稿から参照させる画像など（原稿とまとめてPDFにした方がいい可能性もあり）
- デザイン定義書.yaml
- レイアウト定義書.yaml
- レイアウトリファレンス.pdf（画像をまとめたもの）

これらから、原稿にデザインの要素も追加した「スライド設計書」を作成します。ここまではまだテキストです。
このスライド設計書を作り込むことがスライドの完成度を高めることになります。一般的な「スライドを作る」に相当する作業がここになります。
なお、スライド設計書とはどんなものか？がわかるサンプルがないとAIは何を作ればよいのかわからないので、上記に加えて

- スライド設計書サンプル.md

もインプットとして用意しておきます。

スライド設計書サンプルの例

```md
# スライド設計図（構成案）

※これはAIが出力する設計図のサンプルです。実際の出力ではありません。

---
## スライド1: タイトル（表紙）

* **適用レイアウトパターンID:** `S01` (タイトルスライド・表紙)
* **レイアウト選定理由:**
    * プレゼンテーションの開始であり、タイトルを最も強調して伝えるため。
* **画面構成と配置:**
    * 背景はクリーンな白（#FFFFFF）。
    * 画面中央よりやや上にメインタイトルを最大フォントで配置。
    * その下にサブタイトルを配置。
    * 画面右下に、日付と発表者情報を小さく配置。
* **配置要素の詳細指示:**
    * **メインタイトル:** 「2025年度 新業務基盤導入計画」
    * **サブタイトル:** 「AI活用による工数削減とDX推進に向けて」
    * **右下情報:** 「2025年4月1日 / 株式会社サンプル DX推進部」

---
## スライド2: アジェンダ（目次）

* **適用レイアウトパターンID:** `S02` (アジェンダスライド・目次)
* **レイアウト選定理由:**
    * プレゼンテーションの全体像と流れを最初に共有するため。
* **画面構成と配置:**
    * 画面上部にスライドタイトル「本日のアジェンダ」を配置。
    * その下に、大きな番号付きリストで各セクションのタイトルを縦に並べる。
* **配置要素の詳細指示:**
    * **リスト項目1:** 「1. 導入の背景（現状の課題）」
    * **リスト項目2:** 「2. 新業務フローの概要」
    * **リスト項目3:** 「3. 期待される導入効果（まとめ）」
```

### Step 2: 定義書通りにドラフト版を生成する
ここではAIに「作業者」として、指示通りにスライドを作ってもらうことを任せます。
ここまでで以下のインプットが揃います。

![](./ignore/input.png)

- 原稿.md
- 原稿から参照させる画像など（原稿とまとめてPDFにした方がいい可能性もあり）
- デザイン定義書.yaml
- レイアウト定義書.yaml
- レイアウトリファレンス.pdf（画像をまとめたもの）
- スライド設計書.md

これだけあれば、AIはかなりの精度で意図通りのスライドを作ってくれるでしょう。
もちろんデザインに1つの正解は存在しないので毎回多少のブレは発生しますが、70%～90%程度の完成度のドラフト版を毎回生成してくれると思います。
後述しますが、**ここでAIにどんなアウトプットを出させるかが、次の「人間による最終調整」のやりやすさに大きく影響します**

### Step 3: 人間による最終調整をする
AIが作成したドラフト版から完成版を作成します。
ここはStep2でどんなアウトプットを出したかによってどのように調整をしていくのかが大きく変わります。
例えば画像としてアウトプットした場合は画像を編集する必要がありますし、PPTX形式でアウトプットした場合はPowerPointで編集していくことになります。
